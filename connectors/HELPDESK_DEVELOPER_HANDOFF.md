# Help Desk Developer Handoff

Give this file to the developer or to their AI coding agent together with the platform-specific guide.

## Goal

Add the Switch&Save internal Help Desk to the customer app without uploading the customer database.

The app sends:

- safe software map: modules, screens, menu paths, steps, fields, errors
- action manifest: action names, required fields, risk, confirmation rules, and optional local role hints
- small action results when staff asks for live data

The app does not send:

- full product/customer/invoice/order tables
- database passwords
- connector token in public pages
- raw SQL generated by an LLM

## What The Developer Must Inspect

Ask the developer or AI agent to inspect:

- routes/screens/pages/forms
- sidebar/menu definitions
- role/permission checks
- product, stock, customer, order, invoice, purchase, and report services
- validation schemas and error messages
- navigation APIs or deep links
- background job/queue setup

## Discover The Real Menu First

The downloaded connector contains starter examples only. Before syncing, replace
the sample screens with a discovered map of the real POS/admin app.

Ask the AI agent to scan every route/menu/form/page and generate one screen doc
for each staff workflow, including common POS flows such as:

- Dashboard and main menu
- Products list and add/edit product
- Stock check, low stock, stock adjustment, and stock history
- Orders list, create order, order details, and order status
- Customers list, create/edit customer, and customer history
- Purchase orders and supplier screens
- Invoices, payments, refunds, and receipts
- Daily sales, end-of-day, stock value, and cashier reports
- Settings, printer, terminal, sync, tax, staff, and branch setup

Then test every `navigation.routeId` locally before Sync. If a route cannot be
opened yet, keep it as review text but do not mark it as a verified navigation
target.

## Build These Pieces

1. Platform details file
   - Edit `HelpdeskAndroidAppDetails.kt`, `HelpdeskDotnetAppDetails.cs`, or `HelpdeskWebAppDetails.js` first.
   - This is where real screens/pages/forms, fields, routes, actions, and service mappings live.

2. Software map builder
   - Converts app screens into connector documents.

3. Action registry
   - Maps approved action names to real app services.

4. Navigation registry
   - Maps route IDs to app navigation.

5. Embedded staff chat
   - Shows where the customer app mounts it, unless disabled or blocked by route.
   - Uses the default card design with Chat/History tabs, greeting, quick questions, category chips, large input, and settings.

6. Preview/audit/sync
   - Lets developer/admin check what will be sent before sync.

7. Route verification
   - Tests each `routeId` locally before sync.

8. Worker/WebSocket
   - WebSocket first when available.
   - Polling fallback when WebSocket is unavailable.

## Standard Flow

```text
Developer maps screens/actions
Connector previews/audits manifest
Connector syncs safe docs/actions
Admin reviews docs/actions
Staff opens Help Desk chat inside app
Staff sees default card design
Bot answers from approved docs
Bot shows navigation buttons or guided actions
App opens routes locally or runs local handlers
```

## Chatbox Visibility

Company admin controls visibility in Switch&Save:

```text
enabled
show mode: floating / embedded / hidden
optional route targeting
blocked routes
auto-open
position
```

Developer enforces it inside the app:

```text
if helpdeskChat.shouldShow(currentRoute, currentRole):
    render chat
else:
    hide chat
```

Chat is not blocked by staff role. Staff role still flows through requests for
audit and local action permission checks. Blocked routes always win.

## Navigation Buttons

If bot returns:

```json
{
  "label": "Open Create Purchase Order",
  "routeId": "purchase_orders.create"
}
```

The app handles it locally:

- Android: `navController.navigate(...)`
- .NET: `OpenPurchaseOrderScreen()`
- Web/Node/Laravel: router redirect

The platform never directly controls local screens.

## Simple MVP Safety Model

Do not build a heavy permission matrix first. Switch&Save company admin controls:

- Help Desk enabled/disabled
- allowed and blocked screens
- default and connector-generated quick pills
- action enabled/disabled
- confirmation required for read/report actions if desired

For create/update/danger actions, confirmation is always required. The connected app may still keep its own local role checks, but Switch&Save does not require a complex role setup for MVP.

## Guided Actions

For write/update actions:

```text
Bot asks for fields
UI shows form
Staff confirms exact change
Connector runs local handler
Result returns to bot
```

Never run write/update/danger actions without one explicit confirmation. Use dry-run for sandbox testing.

## Minimum First Actions

Start with:

- `search_product`
- `get_product`
- `check_stock`
- `low_stock_products`
- `daily_sales_report`
- `end_of_day_report`
- `update_product_price`
- `update_product_quantity`
- `create_purchase_order` if the app supports supplier purchase orders

## Done Checklist

- [ ] Connector token stored securely.
- [ ] Software map previews correctly.
- [ ] Audit has no blockers.
- [ ] Docs/actions sync.
- [ ] Staff chat appears only on staff screens and never on blocked/public/customer routes.
- [ ] Customer/public pages never show Help Desk chat.
- [ ] Navigation buttons open correct screens.
- [ ] Read/report actions return small safe JSON.
- [ ] Write actions require explicit confirmation.
- [ ] Dry-run sandbox action works before real writes.
- [ ] Help Desk audit logs show chat, confirmation, action, and result.
- [ ] WebSocket connects or polling fallback works.
- [ ] Health logs appear in Switch&Save dashboard.
